Calamus Editor Spec Review

Calamus의 공개 번들과 현재 브라우저 DOM을 기반으로 에디터 구조를 추적한 메모입니다. 목적은 Celion이 어떤 아이디어를 가져오고, 어떤 복잡도는 피해야 하는지 판단하는 것입니다.

작성일: 2026-05-18 대상: calamus.design editor 근거: live DOM + public JS bundle 범위: editor architecture

1. 결론 먼저

렌더링은 HTML/CSS라기보다 SVG

앱 UI는 React/HTML/CSS지만, 편집 대상 슬라이드는 inline SVG가 중심입니다.

수정은 command log 방식

baseSvg + draftCommands로 미리보기 SVG를 재생성하고 저장합니다.

Celion은 일부만 가져오는 게 맞음

전체 벡터 에디터가 아니라 stable id, command, hash guard가 핵심입니다.

한 줄 판단: Calamus는 “HTML/CSS 기반 문서 편집기”가 아니라 “React 앱 안에 들어간 SVG scene editor”에 가깝습니다. AI가 만든 결과물이더라도, 최종 편집 단위는 HTML section이 아니라 SVG node와 editor command입니다.

2. 이게 HTML/CSS로 만든 게 맞을까?

반은 맞고, 핵심은 아닙니다. 서비스의 껍데기, 툴바, 사이드바, 인스펙터, 버튼, 라우팅은 일반적인 React SPA의 HTML/CSS로 만들어져 있습니다. 하지만 사용자가 실제로 편집하는 슬라이드 본문은 HTML/CSS 레이아웃이 아니라 <svg>입니다.

영역 기술로 보이는 것 판단
앱 shell Vite + React SPA, CSS bundle HTML/CSS UI가 맞습니다.
편집 캔버스 inline SVG, viewBox="0 0 1600 900" HTML/CSS 문서가 아니라 SVG 장면입니다.
선택/리사이즈 UI SVG node metadata + overlay handles SVG 위에 editor overlay를 얹은 구조입니다.
AI 생성 결과 SVG markup + layout metadata AI가 HTML/CSS 페이지보다 SVG 슬라이드를 생성하는 쪽으로 보입니다.
중요한 차이: HTML/CSS는 문서 흐름, 반응형, 긴 글 편집에 강합니다. SVG는 고정 비율, 정밀 배치, 프레젠테이션/이미지 export에 강합니다. Calamus는 후자에 최적화된 선택으로 보입니다.

3. 확인된 전체 스택

분류 확인된 흔적 해석
Frontend Vite, React, React Router, modulepreload chunks Next.js가 아니라 Vite 기반 React SPA로 보입니다.
Auth Supabase auth session, auth state listener 로그인/세션은 Supabase Auth 기반으로 보입니다.
Server API VITE_API_URL, Railway-hosted API URL 프론트와 API 서버가 분리된 구조입니다.
Data fetching TanStack Query queryClient, invalidateQueries deck/runtime-state/bootstrap 데이터를 query cache로 관리합니다.
State Zustand-style store traces 에디터/워크스페이스 상태가 client store에 강하게 묶여 있습니다.
Observability Sentry, PostHog, GTM/GA 사용자 행동과 에러 추적이 들어가 있습니다.

4. 에디터의 핵심 구조

1. Server slide 서버에서 slide SVG 또는 deck data를 받아옵니다.
2. Parse SVG DOMParser로 SVG를 파싱하고 node list, nodeMap, bbox, textContent를 만듭니다.
3. Normalize scene 노드마다 data-editor-node-id, data-editor-node-type을 부여하고 baseSvgHash를 계산합니다.
4. Edit as commands 사용자 수정은 live DOM에 바로 박는 대신 draftCommands 배열로 쌓습니다.
5. Replay preview base SVG + commands를 재생해서 previewSvg를 만들고 화면에 보여줍니다.
6. Save editedSvg, commands, baseHash, editedHash를 함께 저장합니다. 
{
  baseSvgHash: string,
  baseScene: Scene,
  scene: Scene,
  previewSvg: string,
  selectedNodeIds: string[],
  selectionScopePath: string[],
  activeTextRangeSelection: Range | null,
  draftCommands: EditorCommand[],
  historyPast: EditorCommand[][],
  historyFuture: EditorCommand[][],
  canUndo: boolean,
  canRedo: boolean,
  isSlideDirty: boolean
}

5. 노드와 선택 모델

Calamus는 SVG를 그냥 그림으로 취급하지 않습니다. 각 SVG 요소를 편집 가능한 node로 해석하고, 선택/이동/인스펙터를 안정적으로 연결하기 위한 metadata를 적극적으로 붙입니다.

확인된 노드 metadata

  • data-editor-node-id: 편집기 내부의 stable node id
  • data-editor-node-type: group, text, tspan, image, shape, other
  • data-block-id: AI layout block과 연결되는 의미 단위
  • data-editor-select-role: container/leaf 등 선택 역할
  • data-editor-select-behavior: direct/delegate/scope 계열 선택 동작
  • data-editor-select-container-kind: generic, slot 등 container 성격

node id 우선순위

  1. 이미 있는 data-editor-node-id
  2. data-block-id 기반 block:{id}
  3. SVG의 id 기반 id:{id}
  4. DOM path 기반 fallback path:svg[1]/...
이 설계의 핵심은 “보이는 SVG DOM”과 “편집 가능한 의미 단위”를 분리하는 것입니다. 그래서 사용자가 텍스트나 카드 그룹을 클릭했을 때, leaf path가 아니라 block/container 단위로 선택될 수 있습니다.

6. Layout metadata

슬라이드 SVG 안에는 <metadata id="layout-plan">가 있고, 그 안에 block bounding box와 검증 정보가 JSON으로 들어갑니다. 

{
  "blocks": [
    { "id": "title-main", "x": 200, "y": 80, "w": 1200, "h": 180 },
    { "id": "hero-illustration", "x": 444, "y": 290, "w": 712, "h": 400 },
    { "id": "vision-bullets", "x": 200, "y": 730, "w": 1200, "h": 130 }
  ],
  "checks": {
    "overlap": false,
    "alignment": true,
    "slots_inset": true
  }
}

이 metadata는 선택 박스, 인스펙터 좌표, block 단위 조작의 근거로 쓰이는 것으로 보입니다. metadata bbox가 없으면 rect/image/circle/line/text/path 등 SVG 요소별 fallback bbox 계산을 합니다.

7. Command system

에디터의 가장 중요한 부분입니다. Calamus는 사용자의 편집을 “SVG 문자열 전체 교체”로만 다루지 않고, 재생 가능한 command로 기록합니다.

Command kind 역할
setText 텍스트 노드의 문자열을 교체합니다.
setStyle 선택 노드의 style/attribute patch를 적용합니다.
setTextRangeStyle 텍스트 일부 range에만 스타일을 적용합니다.
setGeometry x/y/width/height/cx/cy/x1/y1/x2/y2 같은 geometry 값을 바꿉니다.
transform translate, scale, rotate를 transform으로 적용합니다.
insertImage 업로드된 asset을 SVG <image>로 삽입합니다.
insertFragment 텍스트 박스, 도형, 복제 fragment 등을 SVG markup으로 삽입합니다.
deleteNodes 선택 노드를 제거합니다.
duplicateNodes 선택 노드를 복제하고 offset을 줍니다.
groupNodes / ungroupNodes 복수 노드를 사용자 그룹으로 묶거나 풉니다.
reorderNodes front/back/forward/backward 레이어 순서를 조정합니다.
인접 command는 normalize/coalesce됩니다. 예를 들어 같은 노드의 연속 text/style/geometry/transform 수정은 history와 저장 payload가 불필요하게 커지지 않도록 합쳐지는 흐름이 보입니다.

8. 저장 모델

저장은 /workspace/save로 들어가며, 수동 편집 슬라이드는 최종 SVG와 command log를 함께 보냅니다. 저장 후 content JSON에는 수동 편집 상태가 명시됩니다. 

{
  projectId,
  threadId,
  deckId,
  clientSaveId,
  slideId,
  slideOrder,
  baseSvgHash,
  editedSvg,
  commands,
  embeddedAssets,
  entries
}
{
  kind: "manual_edited",
  format: "svg",
  mimeType: "image/svg+xml",
  svg: editedSvg,
  manualEdit: {
    savedAt,
    sourceDeckId,
    sourceSlideId,
    sourceSvgHash,
    editedSvgHash,
    commands
  }
}
이 구조는 “최종 결과물”과 “어떤 원본 위에서 어떤 편집을 했는지”를 같이 보존합니다. 나중에 충돌 감지, 복원, 재생성, history diff를 다루기 쉬운 설계입니다.

9. UI 기능 범위

  • 슬라이드 캔버스: 1600x900 SVG를 viewport에 맞춰 scale
  • 선택 박스: oriented bounding box overlay와 resize handles
  • 플로팅 툴바: 텍스트 편집, 앞/뒤 순서, 삭제 등
  • 인스펙터: x/y/width/height, 텍스트, 스타일, 컴포넌트 추출 UI
  • 삽입 도구: text box, image, rect, line, arrow, ellipse, polygon, star
  • 히스토리: undo/redo, dirty state, pending inline text draft
  • 내보내기: PDF/PPTX/ZIP 관련 endpoint와 dynamic chunk 흔적

10. Celion에 가져올 것과 피할 것

가져올 것

  • 편집 가능한 요소마다 stable id 부여
  • 원본 문서와 preview/export 산출물 분리
  • 수정 사항을 command 또는 override로 저장
  • base hash와 edited hash로 저장 무결성 확인
  • undo/redo를 DOM snapshot이 아니라 command history로 관리

피할 것

  • 처음부터 full vector editor 구현
  • group/ungroup, rotate, text range style을 한 번에 도입
  • SVG-only source of truth로 ebook 본문을 잠그는 것
  • 파생 상태를 과도하게 늘리는 것
  • 문서 흐름 편집이 필요한 ebook을 slide editor처럼 다루는 것
Celion은 page-level ebook_document가 이미 source of truth입니다. 따라서 Calamus처럼 SVG 전체를 권위 데이터로 삼기보다, ebook_document + page edits/overrides로 작게 시작하는 편이 맞습니다.

11. Celion용 제안 스펙

바로 적용한다면 아래 정도가 현실적인 1차 범위입니다. Calamus의 강점을 가져오되, ebook 편집기에 맞게 단순화한 버전입니다. 

type PageEditCommand =
  | { kind: "setText"; nodeId: string; value: string }
  | { kind: "setStyle"; nodeId: string; patch: StylePatch }
  | { kind: "setGeometry"; nodeId: string; values: GeometryPatch; mode: "delta" | "absolute" }
  | { kind: "insertImage"; nodeId: string; assetId: string; pageId: string; src: string }
  | { kind: "deleteNode"; nodeId: string };

type EditablePageState = {
  pageId: string;
  baseDocumentHash: string;
  commands: PageEditCommand[];
  editedHtml?: string;
  editedCss?: string;
  updatedAt: string;
};

이 정도면 Celion은 현재 HTML/CSS 기반 preview를 유지하면서도, 사용자의 직접 편집을 추적 가능한 구조로 보존할 수 있습니다.

12. 아직 확정하지 못한 부분

  • 서버가 editedSvg와 commands 중 무엇을 최종 권위 데이터로 보는지는 확정할 수 없습니다.
  • layout-plan metadata가 AI 생성 단계에서 나오는지, 프론트 런타임이 보강하는지는 확정되지 않았습니다.
  • 실제 저장 API의 내부 DB schema는 공개 번들만으로는 알 수 없습니다.
  • 협업 편집이나 다중 사용자 충돌 해결까지 지원하는지는 확인하지 못했습니다.
이 문서는 Calamus의 공개 클라이언트 코드와 현재 브라우저 DOM 관찰을 기반으로 한 기술 추정입니다. 내부 서버 구현, DB schema, 비공개 prompt contract는 확정 정보가 아닙니다.
