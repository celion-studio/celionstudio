MVP 단계에서 가장 흔한 실패 원인은 모델의 지능 부족이 아니다. 사용자가 결과를 기다리다 이탈하고, API 비용이 예상치를 넘기고, 30초가 넘는 응답에 프론트엔드가 멈추는 것이다.

3초
응답 지연 1초마다
이탈률은 약 7% 증가한다

비싼 모델일수록 latency가 길고, 긴 응답은 실패 확률을 높인다. MVP의 적은 품질 문제가 아니라 시스템 안정성 문제다.

* GPT-4o 기준 평균 응답 시간은 3–8초. 긴 컨텍스트나 복잡한 추론이 들어가면 15–30초까지 늘어난다.

“비싼 모델에 긴 결과물을 한 번에 요청하는 것은 마라톤을 전력 질주로 뛰는 것과 같다. 초반에 지쳐 넘어질 확률이 높다.”

출력 토큰이 길어질수록 latency는 선형 이상으로 증가하고, 중간에 hallucination이나 구조 이탈이 발생할 확률도 함께 올라간다.

프레임워크

모델 선택 기준은
“품질”이 아니라
“실패 비용”이다

실패 비용 낮음

결과가 틀려도 사용자 경험에 치명적이지 않은 작업. 내부 초안, 아이디어 브레인스토밍, 형식 변환 등.

→ 싼 모델로 충분
실패 비용 높음

결과가 사용자에게 바로 노출되고, 오류가 신뢰를 무너뜨리는 작업. 최종 출력, 고객 응대, 결제 관련 텍스트 등.

→ 고급 모델 검토
핵심 원칙: 모델을 고르는 기준은 얼마나 똑똑한가가 아니라 실패했을 때 얼마나 큰 손해인가이다.

비용 관점

단계별로 모델을 다르게 쓴다

01
Planning
구조 설계, 섹션 분할, 프롬프트 체인 설계
→ Cheap Model
02
Drafting
Batch 단위 초안 생성, 구조 채우기
→ Strong Renderer
03
Rendering
본문 생성, 포맷팅, 톤 일관성 확보
→ Strong Renderer
04
Polishing
최종 검토, 구조 이탈 교정, edge case 처리
→ Premium Fallback

비용 비율은 대략 planning 5% : drafting 60% : rendering 25% : polishing 10%. Premium model은 전체 토큰의 10% 이내에서만 사용한다.

판단 기준

MVP에서
Premium Model
써도 되는가?

써도 되는 경우
  • 결과가 사용자에게 최종 노출되는 구간
  • 이전 단계에서 cheap/strong 모델이 실패한 fallback
  • 전체 결과물의 톤·일관성 교정 (조립 단계)
  • 토큰 수가 적고 latency가 짧은 작업
  • 오류 발생 시 비즈니스 손실이 큰 영역
쓰면 안 되는 경우
  • 모든 요청의 기본 모델로 설정
  • 내부 초안·브레인스토밍 용도
  • 긴 컨텍스트를 한 번에 처리
  • 실시간 응답이 필요한 인터랙션
  • 구조가 단순하고 cheap model로도 가능한 작업
엄지 원칙: premium model은 전체 API 호출의 5–10%를 넘기지 않는다. 넘는다면 파이프라인 설계를 의심하라.

MVP에서 중요한 것은 모델의 스펙이 아니라 파이프라인의 복원력이다. 싼 모델로 구조를 잡고, strong renderer로 살을 붙이고, premium fallback으로 마지막 광을 내는 3단계 구조가 결국 가장 빠르고, 가장 저렴하고, 가장 안정적이다.
